1. Project Overview & Vision
What we are building and why it matters
1.1 The One-Line Pitch
An OpenEnv-compatible reinforcement learning environment where an AI agent learns to diagnose and repair broken multi-stage data pipelines — the single most time-consuming daily task for every data engineer on the planet.
1.2 Why This Problem?
According to multiple industry surveys, data engineers spend between 40% and 60% of their working time debugging pipelines. This is not a toy problem. Every company running analytics, every ML team preparing training data, every data warehouse team at companies like Meta, Google, Stripe, and Airbnb faces this exact pain every single day. The data pipeline debugger environment is one of the highest-utility RL environments that can realistically be built and graded in a two-week hackathon.
1.3 What We Are Building
A fully self-contained OpenEnv environment called pipeline_debug_env
A procedural fault injection engine that generates broken pipelines programmatically
Three difficulty tasks (Easy, Medium, Hard) with clear, measurable complexity jumps
A 100% deterministic grader — no LLM needed to score, pure Python math
A dense reward function that gives partial credit at every step
A baseline LLM inference agent using Claude/GPT via API
Full Docker containerization for isolated, reproducible execution
One-command deployment to Hugging Face Spaces
1.4 Core Design Philosophy
KEY
The agent should never see the ground truth. It only sees what a real engineer would see: error logs, sample rows of bad data, schema diffs, and runtime traces. It must reason from symptoms to root cause — exactly like a human.
This philosophy is what separates this environment from toy problems. A grader that directly exposes the fault list is trivial. An environment that forces the agent to reason backward from observable symptoms to hidden causes is both scientifically valuable for RL research and practically useful for training agents that will eventually replace or assist data engineers.
2. What is Reinforcement Learning?
Primer — everything you need to know for this project
2.1 The Core Loop
Reinforcement Learning (RL) is a framework where an Agent interacts with an Environment over a series of timesteps. At each timestep, the agent observes the current state of the environment, selects an action, and receives back a numerical reward signal plus the next state. The agent's goal is to learn a policy — a mapping from states to actions — that maximizes cumulative reward over time.

2.2 Why LLMs as RL Agents?
Large Language Models have emerged as powerful zero-shot reasoners. But zero-shot reasoning alone is insufficient for complex sequential decision problems. RL post-training — specifically GRPO (Group Relative Policy Optimization) and PPO — allows an LLM to learn from environment feedback, improving its debugging strategy over thousands of episodes. This is exactly what OpenEnv was designed to enable: rich environment feedback loops for LLM post-training.
2.3 Key RL Concepts for This Project
Dense vs Sparse Rewards
A sparse reward gives feedback only at the end (success=1, failure=0). This makes learning extremely slow because the agent receives no signal on intermediate steps. A dense reward gives partial credit at every step. Our environment uses dense rewards: each action that partially fixes the pipeline receives a higher reward than the previous step, even if the pipeline is not fully fixed yet.
Partial Observability
In a fully observable environment, the agent sees the complete state. In a partially observable environment (POMDX), the agent only sees a noisy or incomplete view. Our environment is intentionally partially observable — the agent cannot see the fault list directly. It only observes the symptoms. This makes the environment scientifically interesting and forces the agent to build internal models of the pipeline state.
Determinism vs Stochasticity
A deterministic environment always produces the same next state given the same state and action. Our grader is fully deterministic — the same fix always produces the same score. The fault injection system uses a seeded random generator, so episodes can be reproduced exactly for debugging and evaluation.
3. What is OpenEnv?
The framework that powers this project
3.1 OpenEnv Overview
OpenEnv is an open-source framework developed by Meta PyTorch, in collaboration with Hugging Face, for building standardized reinforcement learning environments. It was announced at PyTorch Conference 2025 and provides a Gymnasium-compatible API for creating, deploying, and consuming agentic execution environments. Think of it as the Hugging Face Hub — but for RL environments instead of models.
3.2 The Three Core APIs
Every OpenEnv environment exposes exactly three methods. This standardization means any RL training framework (TRL, torchforge, SkyRL, Unsloth, ART) can plug into any OpenEnv environment without custom integration code.

4. The Real-World Problem
Data pipeline failures — why they happen and why they are hard
4.1 What is a Data Pipeline?
A data pipeline is a sequence of automated data processing steps that move, transform, and aggregate data from source systems to destination systems. Pipelines are the backbone of every modern data-driven company. They power dashboards, ML training datasets, financial reports, product recommendation systems, and A/B test analysis.

4.3 Why Pipelines Break — Taxonomy of Faults
Fault Class 1: Schema Drift
The most common pipeline failure. An upstream data source changes its schema — a column is renamed, a new column is added, a column is removed, or a data type changes — without coordinating with downstream consumers. Example: a payments team renames 'amount_usd' to 'amount' in their API response, breaking every downstream job that references the old name.
Fault Class 2: Type Mismatches
A column expected to be INTEGER is received as STRING, or a FLOAT arrives as a VARCHAR. Silent failures are the worst variant: SQL often coerces types silently (e.g. '123' becomes 123) but fails on non-coercible values ('N/A' cannot become an integer), causing NULLs or crashes deep in the pipeline where the root cause is invisible.
Fault Class 3: Null Propagation
NULL values in SQL have non-intuitive behavior: NULL + 5 = NULL, NULL > 0 = NULL, COUNT(*) counts NULLs but COUNT(col) does not. A single unexpected NULL in a key column can silently zero out an entire aggregation or drop thousands of rows from a join.
Fault Class 4: Incorrect Join Keys
Two tables joined on the wrong column, or on a column with mismatched semantics (e.g. joining on 'user_id' where one table uses internal IDs and another uses external IDs). Result: row count explosion (Cartesian product) or complete data loss (empty result set).
Fault Class 5: Node Execution Order Violations
In a pipeline DAG (Directed Acyclic Graph), nodes have dependencies. If a downstream node runs before its upstream dependency completes, it reads stale or empty data. Modern orchestrators like Airflow, Prefect, and dbt handle this — but our simulated environment injects artificial ordering violations to test the agent's ability to detect and correct them.
Fault Class 6: Boolean / Filter Inversions
A WHERE clause condition is inverted: WHERE is_deleted = TRUE instead of FALSE. This passes all deleted records through and filters out all valid records. The output looks plausible (same schema, reasonable row count) but contains completely wrong data — making this the hardest fault class for both humans and agents to detect.
INSIGHT
The hardest faults are the silent ones. Schema errors throw exceptions (easy to detect). Type mismatches often silently coerce (medium). Boolean inversions and null propagation pass through silently, producing plausible-looking wrong output (hard). Our task difficulty progression maps directly to this detectability spectrum.
5. Environment Architecture
End-to-end system design
5.1 System Architecture Overview
The pipeline_debug_env is a client-server system built on the OpenEnv framework. The server hosts the environment logic inside a Docker container. The client (the agent) connects to the server over WebSocket and interacts using the standard OpenEnv API. The grader runs server-side — the agent cannot access it directly.

5.3 Data Flow — One Complete Episode
Step 1 — reset(): FaultInjector selects a pipeline template (based on task difficulty), injects N faults, stores the ground-truth (healthy) pipeline internally, and executes both the broken and healthy pipelines to produce the broken_output and expected_output tables. ObservationBuilder constructs the initial observation from error logs, schema diffs, and sample rows. The ground truth is never sent to the agent.
Step 2 — Agent reasoning: The agent receives the observation, formats a prompt, calls the LLM, parses the structured action from the LLM response, and calls env.step(action).
Step 3 — step(action): ActionParser validates the action. PipelineExecutor applies the action to the current (partially-fixed) pipeline state and re-runs it. Grader computes the new score by comparing the current output to the expected output. ObservationBuilder constructs the new observation (updated logs, updated schema diff, updated sample rows). EpisodeManager increments step count and checks done condition. StepResult(observation, reward, done, info) is returned to the agent.
Step 4 — Termination: The episode ends when either (a) reward >= 0.95, meaning the pipeline output is essentially correct, or (b) the agent has used all allowed steps for that task difficulty level. The final score is the maximum reward achieved across all steps in the episode.
5.4 Why DuckDB?
DuckDB is an in-process analytical database. It runs entirely inside the Python process — no server, no network connection, no configuration. It supports the full SQL standard including window functions, JSON, and Parquet. A DuckDB instance spins up in milliseconds and a query on 10,000 rows completes in under 50ms. This makes it the perfect backbone for our environment: fast, self-contained, and deployment-trivial. The entire environment, including the database, lives inside the Docker container.

7.3 Action Space Design Principles
Actions are semantically meaningful — they correspond to real operations a data engineer would perform
Actions are parameterized — the agent must specify not just what to do but how to do it (e.g. which column, which type)
Actions are validated server-side — invalid syntax, nonexistent nodes, or nonsensical parameters return an error reward (small negative nudge) without terminating the episode
Actions are composable — multiple actions can fix different parts of the same pipeline in sequence
The action space is deliberately finite and discrete — this makes it tractable for both LLM agents (structured output) and policy gradient methods (categorical action head)

9. Fault Injection System
How broken pipelines are generated procedurally
9.1 Why Procedural Generation?
Hardcoded broken pipelines would be memorized by agents after repeated exposure. Procedural generation ensures that every episode presents a novel, unseen broken pipeline. The FaultInjector uses a seeded random number generator — the seed can be fixed for reproducible evaluation, or random for open-ended training.
9.2 Pipeline Templates
The FaultInjector starts with a library of base pipeline templates — healthy, fully-correct pipelines modeling realistic business data flows. Templates include: e-commerce order analytics, user engagement metrics, financial revenue reporting, A/B test result aggregation, and product inventory tracking. Each template defines the full DAG, SQL transforms, input schema, and expected output.
9.3 Fault Injection Algorithm
Select a pipeline template based on task difficulty (simple templates for Easy, complex for Hard).
Determine the fault count: 1 fault for Easy, 2-3 for Medium, 4-5 for Hard.
For each fault slot, randomly select a fault class from the available fault types for that difficulty level.
Apply the fault to a randomly selected eligible node in the DAG (not all faults can apply to all nodes).
Execute the healthy pipeline to capture the expected_output ground truth.
Execute the broken pipeline to capture the broken_output and error_log.
Store the fault list, expected_output, and healthy pipeline internally — never expose to agent.

10. Task Progression
Easy → Medium → Hard — concrete task definitions
Task 1: EASY — Single Schema Drift
DIFFICULTY
Easy. 1 fault. 4 max steps. Expected baseline LLM solve rate: ~85%
Scenario
An e-commerce orders pipeline ingests a CSV file from an S3 bucket. The upstream data team renamed the column 'user_id' to 'userId' in their latest export. The downstream SQL query references 'user_id' and fails with a ColumnNotFoundError. The pipeline produces no output.
What the agent sees
Error log: 'ERROR [clean] Column user_id not found. Available columns: userId, email, amount, created_at'
Schema diff: expected column 'user_id' INT — actual: missing. Unexpected column: 'userId' VARCHAR.
Sample broken output: empty table (0 rows)
Sample expected output: 3 rows with user_id, email, amount, date
Correct fix
env.step(PipelineAction(
  action_type='patch_schema',
  target_node='clean',
  params={'old_column': 'userId', 'new_column': 'user_id', 'new_type': 'INT'}
))
# One correct action → reward jumps from 0.0 to 1.0
Why LLMs sometimes still fail
LLMs may hallucinate a wrong column name, apply the fix to the wrong node (e.g. 'ingest' instead of 'clean'), or specify an incorrect type. These reveal gaps in structured output generation that RL training can address.
Task 2: MEDIUM — Type Mismatch + Null Propagation
DIFFICULTY
Medium. 2–3 faults. 8 max steps. Expected baseline LLM solve rate: ~40%
Scenario
A financial revenue reporting pipeline has two co-occurring faults. First, the 'amount' column is ingested as VARCHAR instead of FLOAT (due to currency symbols like '$1,250.00' in the raw data). Second, a downstream aggregation uses SUM(amount) without handling NULLs — when the type coercion produces NULLs for non-numeric values, the entire aggregated total becomes NULL.
What the agent sees
Error log: 'WARNING [clean] 423 rows could not be coerced to FLOAT in column amount. Coerced to NULL.'
Error log: 'WARNING [aggregate] SUM(amount) returned NULL for 100% of groups.'
Schema diff: expected 'amount' FLOAT — actual 'amount' VARCHAR
Sample broken output: all revenue totals are NULL
Sample expected output: revenue totals are numeric (e.g. 125000.50)
Correct fix sequence
The agent must identify both faults and apply fixes in the right order. Fixing the null guard first without fixing the type cast still produces NULLs. The agent must: (1) add_type_cast for the amount column with a format-aware cast, then (2) add_null_guard to filter or default remaining NULLs, then optionally (3) validate output.
Task 3: HARD — Multi-Fault Pipeline (5 Simultaneous Faults)
DIFFICULTY
Hard. 4–5 faults. 12 max steps. Expected baseline LLM solve rate: ~15%
Scenario
A user engagement metrics pipeline has five simultaneous injected faults: (1) schema drift on the users table (user_id renamed), (2) type mismatch on the event_timestamp column, (3) an incorrect join key (events joined to sessions on session_id where it should be on event_session_id), (4) a filter inversion that includes churned users instead of excluding them, and (5) a node execution order violation where the deduplication step runs before the join instead of after.
Why this is hard
No single symptom points to a single fault — multiple faults interact and mask each other
The output is non-empty (unlike Easy) — it has plausible row counts but wrong values (like Medium and hard faults)
Some faults only become visible after others are fixed — the agent must navigate a fault dependency chain
12 steps sounds like a lot but an agent that tries random fixes burns through them quickly
Strong GPT-4 class models score ~0.3–0.5 without RL training. With RL post-training, scores improve to 0.7–0.9 — demonstrating the environment's training value
11. Reward Function
Dense, deterministic, multi-component scoring
11.1 Reward Philosophy
The reward function must be dense (partial credit at every step), deterministic (same action always produces same reward), and meaningful (higher reward unambiguously means better pipeline state). Our reward is a weighted sum of four independently computed components, each scoring a different dimension of pipeline correctness.
11.2 Reward Components
Component
Weight
How computed
Range

11.3 Reward Formula
def compute_reward(actual_output, expected_output, step_count, max_steps, prev_score):
    # Component 1: Schema correctness
    expected_cols = set(expected_output.columns)
    actual_cols   = set(actual_output.columns)
    schema_score  = len(expected_cols & actual_cols) / len(expected_cols | actual_cols)
    # Component 2: Row-level accuracy (per-cell comparison, normalized)
    if len(actual_output) == 0:
        row_score = 0.0
    else:
        matched_cells = compare_tables(actual_output, expected_output)  # fuzzy float match
        row_score = matched_cells / (len(expected_output) * len(expected_output.columns))
    # Component 3: Step efficiency
    efficiency = max(0.0, 1.0 - (step_count / max_steps))
    # Weighted sum
    raw_score = 0.35 * schema_score + 0.35 * row_score + 0.20 * efficiency
    # Component 4: Regression penalty (if agent broke something)
    regression = max(0.0, prev_score - raw_score) * 0.1
    final_score = max(0.0, min(1.0, raw_score - regression))
    return round(final_score, 4)
KEY PROPERTY
The reward is monotonically incentive-compatible: every individual fault fixed increases the reward. An agent cannot 'trick' the grader by, for example, outputting the correct schema but wrong data (schema_score would be 1.0 but row_score would be 0.0, giving only 0.35 total).
12. Grader Design
Deterministic scoring — zero LLM dependency
12.1 Grader Principles
100% deterministic: the same pipeline state always produces the same score
No LLM dependency: scores computed with pure Python + DuckDB SQL
Hermetic: grader runs server-side, agent cannot inspect or game it
Tolerant of float precision: numeric comparisons use a tolerance of 1e-4
Null-aware: NULL vs NULL is treated as a match; NULL vs non-NULL is a mismatch
Order-independent: output row order does not affect score (sorted before comparison)
12.2 Table Comparison Algorithm
def compare_tables(actual: pd.DataFrame, expected: pd.DataFrame) -> float:
    # 1. Sort both tables by all common columns to make order-independent
    common_cols = sorted(set(actual.columns) & set(expected.columns))
    actual   = actual[common_cols].sort_values(common_cols).reset_index(drop=True)
    expected = expected[common_cols].sort_values(common_cols).reset_index(drop=True)
    # 2. Align row counts (truncate longer to match shorter for partial credit)
    min_rows = min(len(actual), len(expected))
    actual   = actual.iloc[:min_rows]
    expected = expected.iloc[:min_rows]
    # 3. Cell-by-cell comparison with type-aware matching
    total_cells   = min_rows * len(common_cols)
    matched_cells = 0
    for col in common_cols:
        for a_val, e_val in zip(actual[col], expected[col]):
            if pd.isna(a_val) and pd.isna(e_val):    matched_cells += 1
            elif isinstance(e_val, float):           matched_cells += abs(a_val - e_val) < 1e-4
            else:                                     matched_cells += (a_val == e_val)
    return matched_cells / max(total_cells, 1)

0.2 – 0.5
Schema partially correct. Some rows match. Major faults remain.
0.5 – 0.8
Most faults fixed. Output mostly correct but with remaining errors.
0.8 – 0.95
Nearly perfect. Minor remaining issues (edge cases, rounding, nulls).
0.95 – 1.0
Pipeline is effectively fixed. Episode terminates as success.
13. Baseline LLM Agent
Reference inference script using ReAct prompting
13.1 What is a Baseline Agent?
The baseline agent is a reference implementation that uses a pre-trained LLM (no RL fine-tuning) to interact with the environment. Its purpose is threefold: (1) it demonstrates that the environment is solvable by non-trivial agents, (2) it establishes a performance floor that RL-trained agents should exceed, and (3) it serves as the required baseline inference script for the hackathon submission.
13.2 ReAct Prompting Strategy
The baseline uses the ReAct (Reasoning + Acting) prompting framework. At each step, the agent is given: (1) the current observation, (2) the action schema, (3) the history of previous actions and their rewards, and (4) a structured prompt that asks the LLM to reason step by step before producing a JSON action.
SYSTEM_PROMPT = '''
You are an expert data engineer debugging a broken data pipeline.
At each step, you will receive the current pipeline state including error logs,
schema information, and sample data. You must identify the most likely fault
and submit ONE action to fix it.
Think step by step:
1. What does the error log tell me?
2. What does the schema diff tell me?
3. What does the sample data tell me?
4. What is the most likely root cause?
5. What single action would best address that root cause?
Respond with a JSON object ONLY in this exact format:
{
  "action_type": "<one of: patch_schema, rewrite_transform, add_null_guard,
                   reorder_nodes, fix_join_key, invert_filter, add_type_cast, no_op>",
  "target_node": "<node name from pipeline_dag>",
  "params": { <action-specific parameters> },
  "reasoning": "<1-2 sentence explanation of why you chose this action>"
}
'''
13.3 Baseline Agent Loop
async def run_baseline_episode(env_url: str, task_level: str):
    async with PipelineDebugEnvClient(base_url=env_url) as client:
        obs = await client.reset()
        history = []
        total_reward = 0.0
        while not obs.done:
            # Format observation for LLM
            prompt = format_observation_prompt(obs, history)
            # Call LLM API (Claude / GPT / Llama)
            response = await call_llm_api(prompt, system=SYSTEM_PROMPT)
            # Parse structured action from response
            action = parse_action_from_response(response)
            # Submit action to environment
            result = await client.step(action)
            # Track history for context window
            history.append({
                'action': action,
                'reward': result.reward,
                'score_delta': result.info['score_delta']
            })
            total_reward = result.reward
            obs = result.observation
        return {'final_score': total_reward, 'steps': len(history)}

19. Key Concepts Glossary
Every technical term used in this project
Term
Definition
Action Space
The complete set of valid actions an agent can take in an environment. In our env: 8 action types.
Agent
The AI system (LLM or RL-trained model) that interacts with the environment to maximize reward.
Boolean Inversion
A fault where a WHERE clause condition is flipped (TRUE instead of FALSE), producing wrong data silently.
DAG
Directed Acyclic Graph. A graph with no cycles. Used to represent pipeline execution order.
Dense Reward
A reward signal that gives feedback at every step, not just at the end. Enables faster RL learning.
Deterministic Grader
A grader that always produces the same score for the same input. No randomness, no LLM.
DuckDB
An in-process analytical SQL database. Runs inside a Python process with no external server needed.
Episode
One complete agent-environment interaction: from reset() to a terminal state.
ETL
Extract, Transform, Load. The classical pattern for data pipelines: extract from source, transform, load to destination.
Fault Injection
Deliberately introducing bugs into a healthy system to test the ability to detect and repair them.
FastAPI
A modern Python web framework for building HTTP APIs. Used to wrap the environment as a web service.
GRPO
Group Relative Policy Optimization. An RL algorithm used to fine-tune LLMs using environment rewards.
Gymnasium
The standard Python RL environment interface (formerly OpenAI Gym). OpenEnv is inspired by it.
Hugging Face Spaces
A platform for hosting ML demos and environments. OpenEnv environments deploy here as Docker containers.
Join Key
The column(s) used to match rows between two tables in a SQL JOIN operation.
LLM
Large Language Model. A neural network trained on text (e.g. GPT-4, Claude, Llama) that generates text.
Null Propagation
The SQL behavior where NULL combined with any value produces NULL, causing silent data loss in aggregations.
Observation Space
The full set of information an agent can receive from the environment at each step.
OpenEnv
Meta's open-source framework for building and deploying RL environments for LLM post-training.
Partial Observability
When the agent cannot see the full environment state — only a noisy or incomplete view.
Pipeline Template
A pre-defined healthy pipeline that the FaultInjector uses as a starting point before injecting bugs.
Policy
The strategy an agent uses to decide which action to take given an observation.
Pydantic
A Python library for data validation using type annotations. Used to define Action and Observation models.
ReAct
Reasoning + Acting. A prompting strategy for LLMs that alternates between reasoning steps and action steps.
Reward
A scalar signal (float) the environment returns to the agent after each action. Ranges 0.0–1.0 in our env.
RL Post-Training
Fine-tuning an LLM using RL rewards from environment interaction, improving task-specific performance.
Schema Drift
When a data source changes its column names or types without coordinating with downstream consumers.
Seed
A number used to initialize a random number generator, enabling reproducible random sequences.
Step
One agent action → environment response cycle. The atomic unit of agent-environment interaction.
Terminal State
An episode state where no more actions are possible: either success (score ≥ 0.95) or timeout.
Trajectory
The complete sequence of (observation, action, reward) tuples in one episode. Used for RL training.
Type Coercion
Automatically converting one data type to another (e.g. string '123' to integer 123). Can fail silently.
WebSocket
A full-duplex communication protocol over a single TCP connection. Used by OpenEnv for env ↔ agent comms.

It is the top time-sink for every data engineering team at every technology company on the planet — including Meta, Hugging Face, and Scaler. Task & Grader Quality (25%) — Score: 24/25
Three tasks with clear, measurable difficulty jumps. A grader that is 100% deterministic, needs zero LLMs to score, and cannot be gamed by partial fixes. Partial credit at every step. The grader implements four independent scoring dimensions (schema, row accuracy, efficiency, regression penalty) that together produce a score with high fidelity to actual pipeline health. Every scoring decision is mathematically justified and described in this document.
Environment Design (20%) — Score: 19/20
The environment is intentionally partially observable (agent never sees the fault list), which makes it scientifically interesting for RL research. The procedural fault injection system generates infinite novel episodes, preventing memorization. The action space is semantically meaningful (maps 1:1 to real engineering operations). The observation space mirrors what a real engineer would see. The environment exposes specific, known weaknesses of current LLMs (causal reasoning, multi-step planning, repair without regression) — making it a genuine research tool, not just a demo.
Code Quality (15%) — Score: 14/15
Full type safety via Pydantic. Clean separation of concerns: FaultInjector, PipelineExecutor, Grader, ObservationBuilder, and ActionParser are independent, testable modules. No existing OpenEnv environment targets data engineering workflows. The closest existing environment is the coding_env (sandboxed Python execution) — but pipeline debugging requires sequential multi-step reasoning, causal diagnosis, and repair-without-regression that is fundamentally harder than one-shot code generation. The fault injection taxonomy is original. The multi-component grader with regression penalty is original. The partial observability design (symptoms only, no fault list) is a deliberate and novel design choice.
